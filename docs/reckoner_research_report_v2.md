# Reckoner Design Research: Recommendations on 24 Open Questions

**Revision 2** — incorporates the infer-then-fortify testing workflow (formulas inferred from observed data, fortified with original inputs + synthetic regression data). Changes concentrated in Workstream D (new preamble, rewritten D4/D5, amended D1), with corresponding updates to key findings, phasing, and caveats.

## TL;DR
- **Pick the boring, well-benchmarked options for the load-bearing questions:** explicit dependency declaration (A2), a build-system-style suspending scheduler with hashed early cutoff (B1), an egress-fixing allowlist proxy that validates the *resolved IP after every redirect* (E1), a minimal-core JS formula API with a small shaping stdlib (A1), and a corpus-driven catalog of ~10 chart forms (F1). These are the areas where prior art is decisive.
- **Reckoner's mainline testing workflow is infer-then-fortify, not engineer-authored unit tests** — and that workflow is structurally circular unless the platform breaks the circle with test-kind labeling, holdout discipline, and metamorphic relations (see the Workstream D preamble). Fixture capture with visible tier consequences moves from edge case to mainline path.
- **The genuinely uncertain questions are the LLM-quality ones (F4, F2, D5, A1's correctness axis) and the streaming envelope knee (C1).** The evidence says: build the judge-agreement harness *before* trusting any generation-quality number; expect critique-and-revise to help only when it has an external check; and instrument the postMessage/OPFS boundary before committing to a message bus.
- **One premise deserves a flag:** pure-client-side calc is fine for report-scale data but has a documented scale ceiling, and sandboxed/opaque-origin iframes cannot easily reach `crossOriginIsolated`, so SharedArrayBuffer-based parallelism (B3) is architecturally fragile and should not be on the critical path for v1.

## Key Findings
1. **SQL vs. dataframe vs. plain-JS correctness is closer than folklore suggests, but auditability and API-stability break the tie toward a minimal JS core with a small typed shaping library.** Text-to-SQL tops out around 72–82% execution accuracy on BIRD (hard, real databases) and ~86% on Spider; "realistic" review-based scoring reaches ~95%. That is strong for ad-hoc querying but not obviously better than plain code for auditable, diffable, unit-tested formulas — and BIRD's own binary execution metric "only agrees with human experts 62% of the time," so both SQL's apparent errors and its apparent successes are partly scoring artifact.
2. **Explicit dependency declaration is the safer default; traced/recorded dependencies (MobX/Incremental/Salsa) are an optimization, not a foundation.** The confidentiality model ("undeclared reads are unnameable") *requires* explicit declaration for its security guarantee; dynamic dependencies are handled via parameterized/indirect input cells, exactly as build systems handle dynamic deps.
3. **The recalculation engine should be a suspending scheduler with verifying (hash) traces and early cutoff** — the "Build Systems à la Carte" design point that Shake and Bazel occupy, which subsumes the spreadsheet dirty-mark+topo-sort approach and adds early cutoff, which Excel notably lacks.
4. **Taint tiering can ride the same traversal as value recompute** (floor/greatest-lower-bound over inputs is a monotone lattice fold), and IFC literature (LIO, COWL) both validates the model and warns about its main failure mode: over-tainting / label creep.
5. **Egress-fixing is a solved problem with a known-correct recipe and a known residual.** Allowlist (not blocklist), resolve-then-pin the IP, re-validate after every redirect (or disable redirects), block RFC1918/link-local/metadata, and route through a single egress proxy. The residual — exfiltration in a request body to a *legitimately* allowlisted host — cannot be closed by egress rules and must be handled by budgets + tiering.
6. **LLM-as-judge is usable but must be calibrated per-dimension against humans first** — agreement can look like ~80% raw yet collapse once chance-corrected (kappa deflation of 33–41 points), and position/verbosity/self-preference biases are real and measurable.
7. **Tests derived from the data a formula was inferred from certify nothing about correctness** — they are characterization (golden-master) tests, green by construction. The correctness mechanisms for the infer-then-fortify workflow are held-out rows (the train/test split, made a first-class affordance) and metamorphic relations (oracle-free invariants like permutation-invariance and conservation), with mutation testing as the offline check on whether the pinned suite has teeth.

## Details

### WORKSTREAM A — Formula API

**RQ-A1 — API surface. Recommendation: (a) minimal JS core over plain values, plus a small typed shaping stdlib (group/join/window/aggregate/pivot). Reject SQL-hybrid as the primary surface; offer a dataframe-style fluent API only as sugar over the same core.**
The decisive axes are auditability, additive-only stability, and LLM first-attempt correctness. On correctness, text-to-SQL is strong but not dominant: state-of-the-art on BIRD (real, messy databases) is 71.83% execution accuracy (Arctic-Text2SQL-R1-32B, Snowflake, May 20 2025 leaderboard), with heavily-engineered agent pipelines reaching 81.67% (Agentar-Scale-SQL, arXiv 2509.24403) and ~86% on the cleaner Spider benchmark. MotherDuck's "Your Data Model Is the Semantic Layer" analysis reports up to 95% accuracy on 500 questions with zero context under review-based scoring — but the same space of evidence undercuts the benchmark itself: the FLEX-metric study (Jin et al., VLDB/CIDR 2026) found BIRD's binary execution accuracy "only agrees with human experts 62% of the time." So much of SQL's apparent error rate is scoring artifact — but so is much of its apparent success. Crucially, BIRD's financial domain has a documented ~49% noise rate and 22 wrong gold queries, so SQL's headline numbers are shakier than they look. For Reckoner, SQL's problems are structural, not accuracy: (i) SQL strings are opaque to the additive-only API contract and to unit-test-beside-formula review; (ii) SQL-over-registered-inputs blurs into a query engine (DuckDB-WASM is heavy). Plain JS with a small stdlib gives the cleanest reviewable diffs and the tightest self-description surface. A dataframe object (Arquero-style) is the right *ergonomic* middle, and the API-assisted code-generation literature shows LLMs drive both Pandas and a pandasql SQL API competently — so offer fluent table methods as a thin, well-documented layer over the same plain-value core rather than as a separate engine.
*Residual uncertainty / experiment:* Run a head-to-head first-attempt-correctness + hallucinated-API-rate eval across three surfaces (plain JS + stdlib, fluent dataframe, SQL-hybrid) on a corpus of ~50 real report shaping tasks, driven only by the API's own self-description. Measure hallucinated-method rate and diff size on a follow-up edit. This is the single most valuable A-workstream experiment.

**RQ-A2 — Dependency declaration. Recommendation: (a) explicit declaration as the foundation; (c) static analysis as a lint/assist; (b) tracing only internally as an optimization if ever needed.**
Explicit declaration is not merely a style choice here — it is load-bearing for the "reach, not egress" confidentiality model: if undeclared reads are unnameable, the evaluator can inject *exactly* the declared inputs and nothing else, making Class B data unreachable by construction. Recorded/traced evaluation (MobX, Vue reactivity, Jane Street Incremental, Salsa, Adapton) is elegant for UI reactivity but makes the reachable set a runtime property, which is exactly what you don't want at a security boundary. Dynamic dependencies (`cells[region]` for a viewer-picked region) are handled the way build systems handle them: parameterize the dependency (the region selector is itself a declared input cell), or use input-cell indirection (a declared "which cell" input whose value names the target within a declared namespace). This is precisely Shake's monadic/dynamic dependencies and Bazel's restarting-scheduler treatment from "Build Systems à la Carte." The cautionary tale is Excel's `INDIRECT()`, which is volatile precisely because its dependency is not statically knowable — recalculated every cycle, defeating incrementality. Reckoner should make the dynamic case explicit (parameterized dependency patterns) rather than volatile.

**RQ-A3 — How feeds appear to formulas. Recommendation: frozen snapshot per recalculation as the default (feed = just another input), with an explicit `window`/frame abstraction as opt-in for feeds where history matters. Reject the query-handle-to-connector option.**
Snapshot-per-recalc is the model that makes purity (A4) and trace-replay debugging (D3) sound: within one evaluation the feed is an immutable value. This mirrors streaming-SQL "materialized view at a point in time" semantics (Materialize, Flink) and Rx snapshotting. The explicit window abstraction (`feed.window('1h')`) is the right escape hatch, and its semantics should follow Flink event-time windowing / Rx `ReplaySubject` buffering rather than being reinvented. The query-handle option is correctly rejected: pushing a query to the connector turns the "dumb pipe" into an agent, violating the connector premise. Excel's RTD/streaming-function model is a useful negative example — it couples volatility to the cell and is hard to reason about; snapshot semantics are cleaner.

**RQ-A4 — Purity. Recommendation: layered defense, classified as a taxonomy. Use SES/Hardened-JavaScript (Agoric) as the environment-starvation + determinism layer, frozen structurally-shared inputs, virtualized nondeterminism (clock/random as declared inputs), and React-strict-mode-style double-evaluation spot checks in authoring/CI only.**
SES is almost purpose-built for this: `lockdown()` freezes shared intrinsics and, with default Compartment options, `Date.now()` and `Math.random()` do not work — removing the two most common ambient-nondeterminism sources exactly as the premise wants. Agoric/MetaMask run it in production (LavaMoat uses a Compartment per module), and it has been through security review with "no critical flaws." Impurity taxonomy:
- *Prevented by construction:* network/filesystem/secrets (environment starvation); mutation of inputs (Object.freeze / structural sharing); ambient clock/random (SES removes them; virtualize as declared inputs).
- *Detected, not prevented:* order-dependence, hidden global state, iteration-order sensitivity → detected by double-evaluation (run twice, compare) à la React strict mode, run in authoring/CI.
- *Accepted residual:* wall-clock performance nondeterminism, floating-point associativity, and infinite loops (SES explicitly "does not protect availability" — any compartment can loop forever), which must be handled by a watchdog/timeout at the worker boundary, not by the language.

**RQ-A5 — API self-description for agents. Recommendation: describe each stdlib function/component as a JSON-Schema-typed tool with a one-line purpose, per-parameter descriptions with formats, enums to make invalid states unrepresentable, and 1–2 worked examples; keep the surface small (well under 20 top-level tools). Gate test: a held-out set of shaping/reporting tasks where the pass bar is a target first-attempt tool-selection + argument-validity rate, with an explicit "irrelevance/abstain" subset.**
The Berkeley Function Calling Leaderboard (BFCL, Patil et al., ICML 2025) is the reference harness: AST-based evaluation over 2,000+ question-function-answer pairs, with explicit relevance/irrelevance ("hallucination") categories. Top models reach ~76–78% overall on the multi-turn BFCL v3 (e.g., GLM-4.5 ~77.8%, Claude Opus 4.7 ~76.6% as of mid-2026) but ~85–88% on single-turn AST — and, tellingly, **irrelevance/abstention accuracy lags badly (~61–79% for GPT-4-class models vs ~85–89% AST)**: models over-call. (Note the BFCL irrelevance metric is itself gameable — a model that never calls functions scores high on abstention — so read it only alongside overall accuracy, per Databricks' "Beyond the Leaderboard" analysis.) Design implications, drawn directly from vendor guidance: Anthropic's "Writing effective tools for agents" (Sept 2025) says prefer *few, high-leverage, namespaced* tools, use search-style over list-all tools, return human-readable context, and iterate descriptions (which "dramatically improves agent performance"); OpenAI's function-calling guide says "use enums and object structure to make invalid states unrepresentable," keep to <20 functions at a turn, and describe *when not* to call. The strongest quantified evidence is Hasan et al., "Model Context Protocol (MCP) Tool Descriptions Are Smelly!" (arXiv 2602.14878), which examined 856 tools across 103 MCP servers: "97.1% of the analyzed tool descriptions contain at least one smell, with 56% failing to state their purpose clearly... augmenting... improves task success rates by a median of 5.85 percentage points... increases the number of execution steps by 67.46% and regresses performance in 16.67% of cases" — so richer descriptions help on net but aren't free. Reckoner should treat the audited catalog's self-descriptions as a first-class, evaluated artifact and include an abstention subset in the gate given the documented over-calling tendency.

### WORKSTREAM B — Recalculation

**RQ-B1 — Recalc algorithm. Recommendation: a suspending scheduler with verifying (content-hash) traces and early cutoff — the Shake/Bazel design point from "Build Systems à la Carte" (Mokhov, Mitchell, Peyton Jones, ICFP 2018). Equality strategy: structural content hashing of immutable snapshots (Unison/Nix-style content addressing) with cheap reference-equality fast path.**
The build-systems taxonomy maps cleanly onto Reckoner. The classic spreadsheet approach (dirty bit + topological sweep; Excel, HyperFormula) is *minimal* but, per the paper, "it's possible but hard to support early cutoff with dirty bits" — and Excel notably does not have early cutoff. Verifying traces (Shake, Ninja) "rebuild a target if the values/hashes of its dependencies changed" and *do* support early cutoff. A suspending scheduler is "theoretically optimal" and, combined with tracking already-built keys, yields minimality; the only caveat is that suspending is better than Bazel's restarting scheduler only when avoided-duplicate-work outweighs suspension cost. HyperFormula is the proof that a JS engine can do dirty-mark + dependency-graph incrementality at scale (stable at 30M+ cells on the NYC Tree dataset; lazy evaluation), but it is GPL-3.0 (commercial license otherwise) and lacks early cutoff, so it's a reference architecture, not a drop-in. Early cutoff needs cheap equality: use immutable snapshots so unchanged sub-results are reference-equal, and content-hash for the "recomputed but identical" case (React.memo-style structural comparison as the semantic model; Unison/Nix content addressing as the hashing model). For async formulas, adopt cancellation-on-restart: a formula whose inputs change mid-flight is cancelled and rescheduled (Bazel's restarting semantics), which is sound because formulas are pure.
*Residual uncertainty / experiment:* Benchmark suspending vs restarting on realistic report graphs (deep chains vs wide fan-out) using something like the js-reactivity-benchmark harness; measure the early-cutoff hit rate on typical edits (the payoff is entirely edit-locality-dependent).

**RQ-B2 — Cycles. Recommendation: endorse "cycles are always an error, with the full cycle path reported; no iterative/fixpoint calc in v1."**
The report-logic tasks named (cohort retention, currency normalization, top-N, period-over-period, sessionization) are all expressible as DAGs — none inherently needs a fixpoint. Genuine fixpoint needs (converging financial models: circular interest/tax, balance-sheet plugs) are real but are a minority even in Excel, where iterative calculation is off by default and switched on deliberately for specific models. Shipping cycles-as-error with a clear diagnostic (full path) matches the majority case and keeps the engine's semantics simple; iterative calc can be an additive, opt-in feature later (consistent with the additive-only rule). HyperFormula's approach — cycle detection via strongly-connected-component decomposition — is the right implementation reference for producing the path.

**RQ-B3 — Parallelism. Recommendation: single evaluator context for v1; design the scheduler so independent subgraphs *could* be partitioned to a worker pool later, but do NOT depend on SharedArrayBuffer. Treat cross-worker data movement as structured-clone/transfer, not shared memory.**
This is the premise most exposed by current browser reality. `crossOriginIsolated` (required for SharedArrayBuffer) needs COOP `same-origin` + COEP `require-corp`/`credentialless`, and critically, sandboxed/opaque-origin iframes are a poor fit: sandboxed iframes without `allow-same-origin` have no storage access, COEP is recursively required of child frames, and developers report opaque-origin sandboxed frames failing to become cross-origin-isolated. So SharedArrayBuffer-based threading inside Reckoner's starved iframe realm is architecturally fragile. The good news is that data movement without SAB is cheap *if you use transfer*: transferring a 32MB ArrayBuffer is ~6.6ms vs ~302ms to structured-clone-copy it (Chrome transferable-objects benchmark); OPFS with a sync access handle in a worker bypasses structured clone for binary entirely. For object graphs, structured clone has a real cost (historically 150ms+ for large objects on Chrome, though modern Chrome removed the worst cliff; stringify-then-postMessage can beat cloning for large JSON). Practical guidance: keep the evaluator single-context for v1; if you parallelize, partition independent subgraphs across workers and move data as transferable ArrayBuffers / columnar binary, not cloned object trees. For reference points, Figma uses WASM threading behind cross-origin isolation on its *own* top-level origin (not a sandboxed child frame), and Google Sheets does heavy calc server-side — neither is a clean precedent for starved sandboxed iframes, which reinforces keeping parallelism off the v1 critical path.

**RQ-B4 — Taint tiering on the recalc graph. Recommendation: yes — propagate tier as a second product of the same traversal (tier = greatest-lower-bound/floor over transitive inputs is a monotone fold over the same DAG). Handle mid-session tier changes as flush-then-restart of the affected subgraph, softened by early cutoff, but never let early cutoff launder a tier.**
Tier propagation is a lattice label fold; value recompute is already a DAG traversal — one pass can carry both, as long as the tier is computed from *input tiers* (not input values), so an unchanged value with a changed tier still re-labels. This is standard lattice-based IFC (LIO's current-label model; COWL's context tainting). The subtlety: early cutoff decides "value unchanged, stop." But a tier can change while a value doesn't (a low-trust feed swapped for another low-trust feed). So the equality check for cutoff must be over the *pair* (value-hash, tier); cutting off on value alone would launder tier. IFC cost evidence is encouraging — Stefan et al., "Protecting Users by Confining JavaScript with COWL" (USENIX OSDI 2014), with implementations in Firefox 31.0a1 and Chromium 31.0.1612.0, "demonstrate a virtually imperceptible increase in page-load latency," and WebKit-bytecode taint tracking is feasible — but the real risk (see E2) is over-tainting, not runtime cost.

### WORKSTREAM C — Streaming / live data

**RQ-C1 — Ingestion envelope. Recommendation: v1 = connector materializes frames to an OPFS-or-IndexedDB mount + a lightweight change notification (BroadcastChannel/postMessage), with binary frames written via an OPFS sync access handle in the connector worker. Trigger a message bus only when the benchmark shows the write-notify-read loop exceeding the freshness budget at target rates.**
The performance facts set the envelope. Structured clone copies everything and is the bottleneck for large frames; transfer/OPFS avoid it (transfer ~6.6ms for 32MB vs ~302ms cloned; OPFS "bypasses the structured clone algorithm for raw binary data" and its sync handle "avoids the async event-loop overhead that accumulates across thousands of small writes"). So the knee is governed by (frame size × whether you copy) and (update rate × per-message overhead). Predicted shape across your 0.1–100 Hz, 1KB–10MB grid: small frames at high rate are dominated by per-message/notification overhead (favor coalescing, cheap notifications); large frames are dominated by copy cost (favor OPFS binary + transfer, avoid clone). Where others put the boundary: the AWS/Bloomberg B-PIPE reference architecture throttles to *one aggregated update per second per ticker* and stores to a queryable store (DynamoDB) — i.e., they conflate at ingestion and read downstream, exactly the materialize-to-mount pattern. Grafana Live and similar push over WebSocket to a client buffer. Recommend: materialize + notify for v1; adopt a bus only if measured end-to-end freshness at, say, 10–30 Hz with mid-size frames blows the budget.
*Experiment refinement:* Measure end-to-end (connector receipt → chart paint) latency, not just postMessage; sweep copy-vs-transfer-vs-OPFS as a third axis; report the 95th percentile (structured-clone cost is tail-heavy).

**RQ-C2 — Windowing/retention. Recommendation: connector owns *raw* retention (keep-last-N / keep-last-T as config, the Kafka-retention analogue); formula-side owns *analytical* windows over whatever the buffer still holds. A chart bound to an expired buffer shows an explicit "data aged out" empty/stale state, never silent wrong data.**
Split ownership matches the streaming canon: the connector is the log with a retention policy (Kafka), the formula's `window()` is the consumer's event-time window (Flink) or `ReplaySubject` buffer (Rx). The buffer must be at least as long as the longest formula window that depends on it — this is a declarable constraint the engine can check. Mobile reconnect / backgrounded-tab is the sharp edge: the Page Lifecycle API (freezing, bfcache) means a backgrounded tab may miss frames entirely; on rejoin the connector should treat it as a fresh subscription with a gap marker, and windows spanning the gap should surface as partial rather than fabricate continuity.

**RQ-C3 — Recompute cadence. Recommendation: coalesce/conflate writes per feed (keep the *recompute* immediate on the coalesced value), with a stated freshness bound; add a fixed display-cadence tick (requestAnimationFrame-aligned) only for the render, not the calc.**
This is precisely the market-data "conflation" practice, which is mature and worth citing: the Bloomberg API exposes `interval=n` conflation ("changes... subject to conflation restriction of at least two seconds between updates") and the AWS B-PIPE design conflates to 1 update/sec/ticker. Conflation (drop intermediate values, keep latest) is the right default because report consumers care about latest state, not every tick. Pair it with React-18-style batching / rAF alignment for paint so you never render faster than the display. State a freshness bound (e.g., "displayed value is at most *X* ms behind the feed"), derived from the conflation interval + one recompute + one paint. Reject naive debounce-the-recompute (adds latency without bounding it); conflate the *input*, compute *once* per coalesced frame.

**RQ-C4 — Freeze semantics as product design. Recommendation: support both paste-values-per-cell (fine-grained) and snapshot-the-workbook (coarse), but make the *tier consequence* explicit and non-surprising at the moment of freezing; generate test fixtures from frozen frames with the frame's tier preserved (no laundering).**
Freezing a live value into a durable one is exactly a lineage/materialization event — the dbt/Dagster "asset materialization" analogue and Observable's cell-pinning. The product risk the premise names is real: freezing a high-tier live value into a personal (lower-tier) sheet is a down-tiering that must be surfaced ("you are copying elevated-trust data into a personal sheet"), because IFC label-lowering is the classic silent-declassification hazard. Excel's paste-values is the ergonomic model users know; borrow its affordance but add the tier annotation. Fixtures derived from frozen frames must carry the source frame's tier so that a test using real captured data cannot be used to launder a tier (ties to D4). Note (revised, see D4): the infer-then-fortify workflow makes fixture-freeze a *mainline* operation, not an edge case — the freeze-with-tier-consequence UX belongs in Phase 2, not Phase 3.

### WORKSTREAM D — Testing / debugging across the sandbox

**The dominant testing workflow: infer-then-fortify (read this before D1–D5)**
Reckoner's testing story is not primarily engineer-authored unit tests. The likely mainline workflow is: *the agent infers a formula from observed data* ("figure out how the numbers in these two columns correlate, write a formula for it"), then *fortifies* it with tests built from the original inputs plus synthetic data for regression. This is two well-studied traditions wearing one workflow, and naming them clarifies what each test kind can and cannot certify:

- The inference step is **programming-by-example** — the FlashFill/Gulwani lineage (Excel's Flash Fill: synthesize from examples, rank candidates by generality). Its documented failure mode is exactly the dangerous one: silently-wrong generalizations that fit the examples perfectly.
- The fortify step is **characterization / golden-master testing** (Michael Feathers' term; the ApprovalTests pattern): pin currently-observed behavior so future changes get flagged. Legitimate and valuable — but it is *regression* evidence, not *correctness* evidence.

The structural trap is **circularity**: if the formula was inferred *from* the original data, tests asserting that it reproduces that data are green by construction; and synthetic regression data generated *by running the fitted formula* is tautologically whatever the formula says. Nobody gamed anything — the workflow itself produces vacuous correctness signals while looking like a well-tested workbook. Jest snapshot testing is the cautionary prior art: pinned outputs decay into rubber when failing snapshots are reflexively updated ("snapshot blindness"). Three mechanisms break the circle, and they reshape D1/D4/D5 below:

1. **Test-kind labeling.** Every test cell declares its kind — *characterization* (pins observed behavior; regression-only weight), *specification* (asserts intended behavior against an independent oracle, e.g. held-out rows or a stated business rule), *property/metamorphic* (asserts an invariant). The review surface and the "is this suite real?" signal weight them differently: a formula covered only by characterization tests derived from its own fitting data is *visibly unvalidated*, the same way an untested cell is visibly untested. This is a cheap, high-leverage addition to D1's structured pass/fail record.
2. **Holdout discipline.** The ML answer to fitting-and-testing-on-the-same-data is the train/test split, and Reckoner can make it a first-class affordance: when the assistant infers a formula from column data, the platform (or the assistant's standing instructions) withholds a slice of rows from the fitting context and uses them as specification tests. Held-out rows are the *only* example-based tests in this workflow that carry genuine correctness weight.
3. **Metamorphic relations** (promoted to a named candidate in D5) — correctness checks that need no oracle at all, which is exactly the situation here: nobody knows the "true" formula, that's why it's being inferred.

**RQ-D1 — Test-execution model. Recommendation: tests are cells (a test is a formula whose value is a structured pass/fail record, now carrying a declared test kind). This is the right default; it inherits incremental re-run from B1 for free.**
Making a test a cell means the dependency machinery already knows which tests to re-run when a formula changes — the same win Bazel test caching and `jest --onlyChanged` provide, but for free from the recalc graph. Wallaby.js-style continuous testing is the UX target. Notably, the infer-then-fortify workflow fits tests-as-cells *better* than traditional unit tests would: a table of (input, expected) pairs is naturally a fixture cell, and the test is a formula over the fixture and the cell under test — so incremental re-run and tier propagation come along for free. The structured pass/fail record should carry the test-kind label from the preamble. The known pitfall to flag (cross-cutting): tests-as-cells means a test can only see what a cell can see (declared inputs) — good for the security model, but it means *integration-style* tests that need to observe cross-cell effects must be expressed as cells over the relevant declared inputs, and flaky/slow tests share the evaluator's watchdog limits. A separate runner realm is unnecessary complexity for v1 and would duplicate the dependency machinery.

**RQ-D2 — Diagnostics out of a silent realm. Recommendation: host-owned, typed, size-bounded, host-tiered structured diagnostic channel, surfaced only in authoring UI; source-map resolution host-side; cap and sample to bound covert-channel bandwidth.**
This is the only design consistent with "realms that must not talk." The pattern is standard structured logging + Sentry-style error boundaries, but with the twist that the diagnostic channel is itself a potential covert channel (a formula could encode Class A data into log volume/timing to leak it). The mitigations are exactly the covert-channel-bandwidth-bounding toolkit: fixed-size typed records, rate caps, sampling, and host-side (not content-side) source-map resolution so the content code never sees the map. Diagnostics inherit the tier of their originating data so authoring UI can suppress what a viewer shouldn't see.

**RQ-D3 — Interactive debugging without DevTools. Recommendation: layer it — (1) rich structured errors + logging (likely 90% of value); (2) trace-based record/replay *outside* the sandbox (sound because formulas are pure); (3) a review-gated "debug evaluator" running the author's own formula with author authority. For agents, prioritize trace-replay over stepping.**
Purity is what makes trace-replay sound: capture declared inputs + intermediates + output, then replay deterministically outside the sandbox (the rr / Replay.io / redux-devtools-time-travel model). This is strictly better for LLM agents than a stepping debugger: agents consume a structured trace (inputs→intermediates→output) far more effectively than they drive an interactive stepper, and replay determinism is guaranteed by A4. The debug-evaluator (author authority) is powerful but is correctly flagged for security review before prototyping, since it relaxes starvation.

**RQ-D4 — Fixture provenance and tiering. Recommendation (revised): fixture capture IS a freeze operation (shares C4's machinery and tier rules) — and in the infer-then-fortify workflow it is the MAINLINE path, not the exception. Real observed data seeds the test suite by default; synthetic data's role is coverage extension, not privacy substitution; and synthetic fixtures must never be generated by running the formula under test.**
The original framing ("synthetic-first, real-data capture as the tier-conscious exception") partially flips once infer-then-fortify is the dominant workflow: "fortify the formula using the original input numbers" means real — possibly sensitive, possibly low-trust — rows become durable document content that ships with the shared workbook, routinely. Consequences:
- *The freeze UX moves up the roadmap.* The moment where the author confirms "these 200 rows are now part of the document, at tier X" is a Phase 2 requirement, not a Phase 3 nicety. It shares C4's machinery: capture inherits the source frame's tier, the tier consequence is shown before confirmation, and down-tiering into a personal sheet is surfaced explicitly (the IFC silent-declassification hazard).
- *Synthetic data gets a different, sharper job description.* Not a privacy stand-in, but **coverage extension**: edge cases the observed data happens not to contain — empty groups, nulls, single-row groups, ties, extreme magnitudes, boundary dates. Generate it from the schema (json-schema-faker-style) or have a *second* agent author it from the cell's stated intent — never by evaluating the fitted formula, which would only pin the formula's own behavior (circularity again). Synthetic fixtures are clean-tier by construction, which keeps the shared suite runnable at low friction.
- *The shareability feature survives and strengthens.* Because fixtures are frozen, tier-tagged data, a viewer can run a shared workbook's full test suite without ever being granted the live feed — the fixtures stand in. With real-data capture as the mainline, this feature does more work, and the tier tag is what makes it safe.
- *Differential privacy on fixtures remains overkill for v1* (assessed and re-affirmed): the tier system already governs whether captured rows can travel with a shared document; DP would add machinery without a matching threat unless fixtures are published broadly at scale.

**RQ-D5 — Making the test loop actually certify correctness (revised from "hard to game"). Recommendation: ship a default loop of (1) test-kind labeling with review-surface weighting (characterization vs. specification vs. property/metamorphic); (2) automatic holdout — infer on a subset, specify on the withheld rows; (3) metamorphic relations and property-based tests in the stdlib; (4) independent test/fixture authoring by a second agent from stated intent; with (5) mutation testing as the offline CI signal that a pinned suite has teeth.**
The reframing matters: the threat is less an agent adversarially gaming tests and more a workflow that is *structurally* circular — but the countermeasures overlap, and the evidence base holds:
- *Characterization tests are decent regression armor and decent mutant-killers.* A mutated formula usually breaks at least some pinned rows, and Just et al., "Are mutants a valid substitute for real faults?" (FSE 2014; 357 real faults, 5 projects, 321 KLOC) found mutant detection correlates with real-fault detection independent of coverage — so mutation score (PIT/Stryker-style) is the cheap offline check on whether a golden-master suite has any discriminating power. Mutation testing gets *stronger*, not weaker, under infer-then-fortify.
- *Metamorphic relations are unusually well-suited and were missing from the original candidate list.* Metamorphic testing was invented for the oracle problem — you cannot assert outputs when no one knows the right answer, but you can assert *relations between runs*: permute the input rows and an aggregate must not change; double every input and a ratio must not change; append a zero-weight row and nothing changes; bucketed totals must sum to the unbucketed total (conservation). These are non-circular (independent of the fitting data), agents state them well (they are the same conservation/invariance properties from the PBT literature), and each relation is just another pure evaluation — cheap under the recalc engine. Promote metamorphic relations to a named stdlib feature beside property-based testing.
- *LLM-authored properties help but do not self-certify.* Vikram, Lemieux, Sunshine, Padhye, "Can Large Language Models Write Good Property-Based Tests?" (arXiv 2307.04346): GPT-4 synthesizes valid+sound PBTs covering only ~20–21% of documented properties; follow-on surveys note LLM-generated properties are "frequently trivial... or incorrect." Tanaka et al. (arXiv 2510.25297): PBT and example tests each caught 68.75% of injected bugs; combined, 81.25% — both belong in the default loop. The Property-Generated Solver (arXiv 2506.18315) reports +9.2 absolute pass@1 using PBT as a validator, while warning that naive automated test generation "can misdirect the correction process."
- *Independent authoring breaks shared context.* A second agent writing specification tests and synthetic fixtures from the cell's *stated intent* (not its implementation, not its fitting data) is the strongest lever against both gaming and circularity — the judge-panel separation pattern.
*Experiment refinement:* the injected-bug corpus should specifically include **plausible mis-inferences** — off-by-one cohort boundaries, wrong join semantics, a formula that fits the observed rows but diverges on unseen ones — and measure catch rate across four arms: characterization-only, +holdout, +metamorphic, full loop. The literature predicts characterization-only performs worst on exactly the mis-inference class (green by construction on fitting data), and the combined-method uplift (~12+ points from the PBT+example evidence) should compound with holdout. Target: ≥80% catch on the full loop, and publish the per-arm breakdown so the review surface's kind-weighting is evidence-based.

### WORKSTREAM E — Security mechanisms

**RQ-E1 — Connector egress-fixing. Recommendation: grant-time target binding enforced by a single host egress proxy that (i) allowlists (never blocklists) scheme+host+port; (ii) resolves the host and pins the IP, re-validating the resolved IP against the allowlist after every redirect hop — or disables redirects entirely; (iii) blocks RFC1918, loopback, link-local 169.254.0.0/16, CGNAT 100.64.0.0/10, IPv6 ULA/mapped, and cloud-metadata IPs; (iv) enforces per-instance fetch budgets (rate/volume) as an anomaly tripwire. Additionally set CSP `connect-src` on the connector frame as browser-native defense-in-depth. Name the residual explicitly.**
This is the OWASP SSRF-prevention recipe, and the literature is unusually unanimous. The core rule: "resolve the URL to its final IP after following all redirects and re-validate that resolved IP against the allowlist — not the URL string." DNS rebinding is the specific trap: any defense that "resolves a hostname, checks the result, and then hands the hostname back to an HTTP library to resolve again is vulnerable." The correct fix is an egress proxy that resolves-and-connects in one step (Stripe's Smokescreen is the named reference implementation), or pinning the resolved IP. Blocklists are explicitly insufficient (decimal/hex/octal IP encodings, IPv6 forms, parser-differential `http://allowed@evil/` bypasses; CVE-2026-27127 in Craft CMS is a recent rebinding recurrence). Wildcard-subdomain grants should be avoided or tightly bounded for the same parser-differential reasons. Prior art for the *grant* model: Deno's `--allow-net=host-list`, Cloudflare Workers outbound rules, GitHub Actions egress policies (StepSecurity Harden-Runner). CSP `connect-src` on the connector frame is a real additional layer the host can set, but is not sufficient alone (it doesn't stop DNS rebinding at the IP layer), hence the proxy. **Named residual: request-body exfiltration to a legitimately allowlisted host.** No egress rule closes this — if the connector may talk to `api.vendor.com`, content-influenced data can ride in a request body there. The per-instance fetch budget is the tripwire (volume/rate anomalies), and tiering (E2) is the containment (content that reached Class B data can't be an input to the fetch).

**RQ-E2 — Output tiering granularity. Recommendation: start with one tier per feed-instance (simplest correct thing); instrument over-tainting; add per-column tiering only when measured over-tainting (one low-trust feed flooring an otherwise-shareable report) exceeds a threshold. Skip per-region tiers unless a specific report pattern demands it.**
The IFC literature's dominant lesson is *label creep / taint explosion*: dynamic IFC systems over-approximate and progressively taint everything, which is the documented reason systems like COWL saw limited adoption and why taint trackers fight false positives. So per-column/per-region tiering is not obviously better — finer labels mean more label-tracking surface and more places to get it wrong. The BI-tool analogue is row-level security in Power BI/Looker, which is powerful but a well-known source of complexity and misconfiguration. Recommendation: ship coarse (per-feed-instance), measure how often the floor rule over-taints a report that users actually wanted to share, and let that number justify column-level granularity. The measurement is the deliverable: instrument "reports where ≥1 element is up-tiered solely due to a single low-trust input that doesn't flow to the shared output."

**RQ-E3 — Aggregate reach view. Recommendation: a persistent reach badge ("9 sources · 2 elevated") plus consent screens that show the *new total*, not just the delta. Validate with a behavioral study, not a preference survey.**
Permission-fatigue research is clear that per-event prompts fail: Felt et al. (SOUPS 2012) showed warning fatigue is worsened by low-value warnings and that users largely don't attend to install-time permissions; runtime-permission field studies (PrivaDroid, USENIX Security 2021) found users are more likely to *deny* unexpected runtime requests — i.e., context and surprise drive decisions more than raw counts. Aggregate dashboards (iOS App Privacy Report, Android Privacy Dashboard) surface totals but the evidence that they *change behavior* is thin — they mostly create awareness. So the design bet is: a persistent aggregate badge (awareness) + a consent moment that frames the *new cumulative reach* (decision point), because the "silent expansion" literature (Android permission-group auto-grant, where "once any group member is granted, later additions are automatically approved") shows the danger is capability creep users never re-consent to.
*Study design:* A/B a delta-only consent vs a new-total consent, measuring grant/deny rates *and* post-hoc comprehension ("how many sources can this report reach?") — behavioral + comprehension, since stated privacy concern notoriously fails to predict grants.

**RQ-E4 — Composite consent on mobile. Recommendation: one-member-per-card progressive disclosure, elevated connectors as their own full-screen step. Endorse the pattern; validate on real devices.**
This is mostly design validation and the pattern is sound: progressive disclosure on small screens reduces cognitive load, and elevating the risky item (the elevated connector) to its own full-screen step mirrors the App Store purchase-confirmation and Android runtime-permission-per-capability patterns that users already parse. The one caveat from the fatigue literature: don't turn every low-risk member into a full prompt (that manufactures fatigue); reserve the full-screen step for genuinely elevated reach. Real-device validation is needed because small-screen truncation and scroll behavior materially change whether users see the aggregate.

**RQ-E5 — One-sentence viewer trust claim. Recommendation: pick a claim that states what is *not* run and names the residual, avoiding the over-claiming that sinks "safe browsing"-style messaging.** Candidates:
1. *"Static reports run none of the author's code in your browser; live reports fetch only from sources you approve, and nothing here can reach your other data."* (Strongest, closest to the architecture; "reach, not egress" made concrete.)
2. *"This report can compute and display data, but it can't access your files, your accounts, or any data you didn't open it with."* (Plain-language capability bound.)
3. *"Viewing is safe by construction: the report's formulas are sandboxed and starved of network and secrets; the only outside data is what its connectors are explicitly allowed to fetch."* (For a more technical audience.)
The over-claiming failure to avoid is the Gatekeeper/"safe browsing" pattern of implying *guaranteed* safety; each candidate is scoped to a true, mechanism-backed claim and (in #1) names the residual boundary rather than promising absolute safety.

### WORKSTREAM F — LLM-generated report templates

**RQ-F1 — Component catalog. Recommendation: a minimal catalog of ~10 forms: (1) KPI/stat card, (2) bar/column (incl. stacked & 100%-stacked), (3) line, (4) area, (5) table/matrix, (6) scatter, (7) histogram, (8) pie/donut *capped at ≤5 slices*, (9) small-multiples/faceting wrapper, (10) map (choropleth/point) — plus text/annotation blocks. Exclude by construction: 3D anything, dual-axis by default, pie beyond 5 slices, radial/gauge except a single KPI gauge, word clouds.**
The visualization literature and BI usage data converge here. Practitioner consensus (ThoughtSpot, Power BI, Databricks docs) is that "the most common chart is the bar chart... line charts come in close second... bar charts tend to dominate" real dashboards, with pie/donut a distant, frequently-misused third. Vega-Lite's grammar-of-graphics and Observable Plot's marks are the right *minimal expressive core* model — a small set of marks composes most dashboards — but Reckoner's closed catalog should ship *chart types*, not a raw grammar, so the agent picks from audited, anti-affordance-reviewed components. The anti-affordances are well documented: 3D pie distorts perception ("slices facing forward look larger"), pies with ≥7 segments are unreadable, and multi-select "% of respondents" data must not be a pie (doesn't sum to 100%). Small-multiples (trellis) is explicitly recommended over cramming series. Corpus-driven refinement (the ~50-dashboard method) should tune the exact cut, but this is a defensible starting catalog.

**RQ-F2 — Message-finding stage. Recommendation: all three, staged — (1) statistical screening tools (trend/outlier/magnitude as *callable, computed* functions so "what changed" is calculated not hallucinated); (2) an agent pass that produces a structured brief (audience, key question, 1–3 findings, supporting breakdowns) which layout must honor; (3) optional author-in-the-loop brief confirmation. Plan-then-generate beats one-shot.**
The automated-insight lineage (Narrative Science/Quill, Tableau Explain Data / Pulse, Power BI Smart Narratives, and research systems DataShot/Calliope/InsightPilot) shows both the value and the failure mode: these tools generate narratives, but the recurring critique is analyst/model *bias* being introduced, and "the end-user may or may not agree." The fix the field converged on is to *compute* the candidate insights (significant increases/decreases, outliers) rather than free-generate them — Tableau Data Stories configures dimensions/measures then surfaces computed insights. So make screening a tool the agent calls, not a thing it guesses. The plan-then-generate structure (brief that layout must honor) is the LLM-pipeline analogue of the same principle and reduces one-shot drift.

**RQ-F3 — Encoding restraint. Recommendation: compose all three, but weight toward (1) catalog ceilings and (2) mechanically-checkable style constraints; use (3) critique-and-revise judiciously and only with an explicit checklist, because self-critique's value is conditional.**
Catalog ceilings (bounded palette, one type scale, no 3D by construction) are the most reliable because they're not subject to model judgment. Of the mechanical constraints, the checkable ones are: max accent colors per view, max tiles per viewport, palette membership, slice-count caps, axis-count caps — all deterministically lintable (vega-lint-style). "Minimum text-to-ink ratio" / Tufte data-ink ratio is *aspirational but hard to mechanize* reliably (operationalizing data-ink has a weak track record); treat it as a judge-pass heuristic, not a hard lint. On critique-and-revise: the evidence is mixed and cautions restraint. Self-refine/Reflexion help most "when initial accuracy is low, question difficulty is high, and *external verification is available*"; without an external check, self-critique can *amplify self-bias* ("Pride and Prejudice: LLM Amplifies Self-Bias in Self-Refinement," arXiv 2402.11436) and the critical survey (TACL, "When Can LLMs Actually Correct Their Own Mistakes?") found *no* prior work showing successful self-correction from prompted-LLM feedback alone outside tasks with reliable external feedback. So the decoration-hunting judge pass should check against the *mechanical rules and the brief* (external anchors), not vibes.

**RQ-F4 — Measuring report design quality. Recommendation: build the anchored rubric (message clarity, hierarchy, chart-form appropriateness, restraint, responsive integrity, theme integrity, accessibility) + a benchmark set, and calibrate LLM judge panels against human experts, measuring judge–human agreement PER DIMENSION FIRST. This harness must exist before any generation-quality claim.**
LLM-as-judge is usable but the reliability caveats are severe and dimension-dependent. Strong judges reach ~80% agreement with humans on open-ended chat — Zheng et al. (MT-Bench/Chatbot Arena, NeurIPS 2023) report GPT-4 judges "match human evaluations at an agreement rate exceeding 80%," the same level as human–human agreement — *but* the systematic biases are real and quantified: "position bias manifests as up to 75% preference for first-positioned responses... self-enhancement bias creates 10-25% preference for self-generated content" (arXiv 2511.04133, summarizing Zheng et al.), plus verbosity bias. Worse, the raw-agreement figure is misleading: "Reliability without Validity" (arXiv 2606.19544; 21 judges, ~541,000 judgments) finds "kappa deflation... universal (33–41 pp on MT-Bench)" and that "judge rankings shift by up to 14 positions across benchmarks" — so an "85% agreement" on MT-Bench corresponds to Cohen's κ≈0.48, and even the best judge (Gemini 3.1 Pro) shows exact-match 0.849 vs κ=0.511 (a 33.8-pp gap). Implications for the protocol: (1) report chance-corrected agreement (Cohen's κ), not raw match, per dimension; (2) use pairwise comparison (more reliable than absolute scoring) where possible; (3) counter position bias with AB/BA order-swapping; (4) put length-neutrality explicitly in the rubric to blunt verbosity bias; (5) avoid self-preference by not judging a model's output with the same model family. The sequencing point is non-negotiable: because judge reliability varies so much by dimension and is non-transferable across benchmarks, the F4 harness (with measured per-dimension κ) must precede any F2/F3 generation-quality claim — otherwise you're optimizing against an uncalibrated ruler.

**RQ-F5 — Responsive. Recommendation: one template with catalog-owned adaptation as the default; allow a per-form-factor *override* only as an escape hatch, not the norm.**
The BI evidence cuts both ways but favors one-template-with-adaptation for an LLM-authored, deterministically-structured artifact. Power BI ships a *separately authored* mobile layout (evidence for the variant approach) while Tableau offers *automatic* device layouts (evidence for adaptation) — the split reflects that hand-authored variants give control at the cost of a second artifact to maintain. For Reckoner, a second authored artifact doubles the diff surface and fights the F6 goal of small reviewable diffs, so catalog-owned adaptation (components know how to reflow; container queries are now mature enough to drive this) is the better default. Reserve per-form-factor variants for the rare report where reflow genuinely can't preserve the message (visualization-retargeting research shows some dense charts don't survive small screens).

**RQ-F6 — Iteration loop / targeted edits. Recommendation: require stable, deterministically-structured, readable template artifacts and drive edits with a search/replace or unified-diff edit format; target ≥90% small-reviewable-diff success. Prefer diff/search-replace over whole-file regeneration for locality, but keep whole-file as the robust fallback.**
Aider's edit-format benchmarks are the reference: models that can use a diff/search-replace format edit larger files far more token-efficiently, and Aider's unified-diff format raised GPT-4-Turbo's score to 59% while making it "3X less lazy," with high-level-diff prompting cutting apply-errors 30–50%. But the nuance matters: Aider also found whole-file is "the easiest format for an LLM to use," and function-call-based edits performed *worse*; a 2025 multi-turn study (arXiv 2510.13859) found whole-file "tends to be more stable overall" across model families even though diff/udiff win on single-turn locality. So: use search/replace or unified diff to get the small reviewable diff you want ("make churn lead, drop the pie" as a localized hunk), lint that it applies cleanly, and fall back to whole-file regeneration when the diff fails to apply. The deterministic structure of the MDX subset (closed component catalog, declarative bindings) is exactly what makes diffs land reliably — unstructured artifacts are where edit-locality breaks down.

## Recommendations (staged)

**Phase 1 — the foundations that gate everything (build these first, in order):**
1. **A2 explicit dependency declaration** → it's the prerequisite for B1 and for the confidentiality guarantee. (Premise dependency: A2 before B1 — endorsed.)
2. **B1 suspending scheduler + hashed early cutoff**, single-context evaluator (defer B3 parallelism).
3. **E1 egress-fixing proxy** with resolve-and-pin + redirect re-validation → gates everything shared; nothing with a live connector should ship before this. (Premise: E1 gates everything shared — endorsed and reinforced by the SSRF literature.)
4. **A4 purity via SES/Hardened JavaScript** + virtualized clock/random → makes A3 snapshots and D3 trace-replay sound.
5. **F4 judge-agreement harness** → must exist before any generation-quality claim (F2/F3). (Premise: F4 before generation-quality claims — endorsed; the kappa-deflation and non-transferability evidence makes this urgent.)

**Phase 2 — the product surface:**
6. A1 minimal-JS-core API + A5 evaluated self-descriptions (run the surface bake-off first).
7. B4 tier-on-the-graph (pair value+tier in the cutoff equality check).
8. **D1 tests-as-cells with test-kind labels + the infer-then-fortify loop (holdout affordance, metamorphic/PBT stdlib) + the C4/D4 fixture-freeze UX with visible tier consequences** — promoted from Phase 3, because fixture capture is the mainline path of the dominant testing workflow, and the test-kind/holdout machinery is what makes agent-inferred formulas reviewable at all.
9. C1/C3 materialize-to-mount + conflation with a stated freshness bound; benchmark the knee before any message bus.
10. F1 catalog + F2 computed message-finding + F6 diff-based iteration.

**Phase 3 — hardening and scale:**
11. E2 measure over-tainting before adding column-level tiers; E3/E4 consent UX behavioral studies; D5's mutation-score CI signal and the four-arm injected-bug study (characterization-only / +holdout / +metamorphic / full loop); B3 worker-pool parallelism *only if* the single-context evaluator misses latency targets and *only* via transfer/OPFS, never assuming SharedArrayBuffer.

**Benchmarks/thresholds that would change these calls:**
- If the A1 surface bake-off shows SQL-hybrid ≥10 points higher first-attempt correctness *and* comparable diff auditability, promote SQL to a co-equal surface.
- If C1 benchmarking shows the write-notify-read loop exceeding the freshness budget at ≤30 Hz mid-size frames, add a message bus in Phase 2 not Phase 3.
- If E2 instrumentation shows >~20% of shareable reports over-tainted by a single non-flowing low-trust input, build column-level tiering.
- If the D5 four-arm study shows holdout+metamorphic catching <60% of plausible mis-inferences, the infer-then-fortify workflow needs a stronger gate (e.g., mandatory second-agent specification tests before a fitted formula can be published to a shared space).
- If B3 becomes necessary and browsers ship reliable `crossOriginIsolated` for credentialless/sandboxed frames, revisit SharedArrayBuffer; until then, transfer/OPFS only.

## Caveats
- **Pure-client-side BI has a documented scale ceiling.** HyperFormula demonstrates 30M+ cells is *possible* in JS, but that's a headless engine on a fast machine; report-scale data is the safe zone and large-data pulls should be pre-aggregated by connectors, not shipped raw to the browser. Flag any report whose pulled frame exceeds a size budget.
- **SharedArrayBuffer/`crossOriginIsolated` in sandboxed/opaque-origin iframes is fragile today** (COEP recursion, no storage access, developer reports of isolation failing) — B3 parallelism must not assume it.
- **Additive-only API evolution** has a known long-tail cost: you can never remove a mis-designed stdlib function, so the A1/A5 surface must be conservative at launch (small is safer than complete). The MCP "smelly descriptions" finding (97.1% of tool descriptions flawed) is a warning that self-descriptions rot; budget for description iteration even under additive-only.
- **Tests-as-cells** inherits the evaluator's watchdog/timeout limits and the declared-input visibility rule — integration tests must be modeled as cells over declared inputs, which is a modeling burden worth documenting.
- **A green test suite is not a correctness claim under infer-then-fortify.** Characterization tests derived from a formula's own fitting data are green by construction; without test-kind labeling, holdout, and metamorphic relations, a workbook can look thoroughly tested while carrying a silently-wrong generalization (the FlashFill failure mode). The review surface must make "validated" and "merely pinned" visually distinct, or the testing story becomes theater.
- **LLM-judge and self-critique numbers are the softest evidence in this report** — treat every F-workstream quality claim as provisional until the F4 harness reports per-dimension chance-corrected agreement. Several supporting figures (BFCL top-line, MCP description study, the 2606.19544 kappa-deflation cohort) come from fast-moving 2025–2026 sources and conflict across versions; the directional findings are robust, the exact numbers are not.
